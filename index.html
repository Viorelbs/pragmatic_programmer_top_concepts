● 01 / CRAFT

Care About
Your Craft

Take responsibility for the quality of what you produce. Leave code cleaner than you found it, ask questions when requirements are vague, and never ship something you know is broken.

⚡
KEY TAKEAWAY
You can only get good at your craft by caring about it every single day not just when someone is watching.

1// Bad: SQL injection risk "it works"
2function getUser(id) {
3 return db.query(
4 "SELECT * FROM users WHERE id=" + id
5 );
6}
7
8// Good: you own this code
9function getUser(id) {
10 if (!Number.isInteger(id)) {
11 throw new Error("Invalid ID")};
12
13 return db.query(
14 "SELECT * FROM users WHERE id = ?",
15 [id]
16 );
17}
● 02 / DRY

Don't Repeat
Yourself

Every piece of knowledge business logic, data structures, documentation should exist in exactly one place. Duplicating knowledge creates a maintenance trap: change it in one place, forget another, bugs appear.

ℹ DRY is about knowledge, not textual similarity. Two similar-looking functions can be fine if they encode different concepts.
⚡
KEY TAKEAWAY
When business rules change (and they will), DRY means you change it once not hunt through 40 files.
discounts.js
1// VIOLATION: logic in 3 places
2price = price * 0.9; // checkout.js
3total = total * 0.9; // cart.js
4amount = amount * 0.9; // invoice.js
5
6// ─────────────────────────────
7
8// DRY: one source of truth
9// config/discounts.js
10export const MEMBER_DISCOUNT = 0.10;
11
12// everywhere else
13import { MEMBER_DISCOUNT } from
14 './config/discounts';
15
16price = price * (1 - MEMBER_DISCOUNT);
● 03 / ARCHITECTURE

Orthogonality

Two components are orthogonal if changing one has zero effect on the other. Your database layer should know nothing about your UI. Your auth module should not care about logging. Non-orthogonal systems are where bugs and fear of refactoring come from.

⚡
KEY TAKEAWAY
If you can change one module without touching any other your system is orthogonal. That's the goal.
profile.js
1// Non-orthogonal: UI knows SQL
2function renderUserProfile(userId) {
3 const user = db.query(
4 "SELECT * FROM users WHERE id=?",
5 [userId]
6 );
7 document.getElementById("name")
8 .textContent = user.name;
9}
10
11// Orthogonal: layers separated
12async function fetchUser(userId) {
13 return api.get(`/users/${userId}`);
14}
15async function renderUserProfile(userId) {
16 const user = await fetchUser(userId);
17 document.getElementById("name")
18 .textContent = user.name;
19}
● 04 / DELIVERY

Tracer Bullets

A thin slice of real, production-quality code that goes end-to-end through the entire system even if empty. Unlike prototypes (which you throw away), tracer code stays. It gives your team something working to react to, early.

⚡
KEY TAKEAWAY
Ship something real that touches every layer first. You'll catch architecture mistakes early, not in month six.
orders.jsx
1// Sprint 1: whole stack, minimal
2
3// Route exists
4app.get('/api/orders', (req, res) => {
5 res.json([]); // empty for now
6});
7
8// UI renders
9function OrderList({ orders }) {
10 return (
11 <ul>
12 {orders.map(o => <li>{o.id}</li>)}
13 </ul>
14 );
15}

18// Now iterate: the skeleton is right.
● 05 / QUALITY

The Broken
Windows Theory

One broken window signals that standards don't matter more hacks follow. When code quality starts degrading, fix broken windows immediately or board them up with a comment and a ticket. Never let rot normalize.

⚡
KEY TAKEAWAY
Code quality is a social contract. Once you let it slip, everyone assumes slipping is acceptable.
cart.js
1// The broken window
2function calculateTotal(items) {
3 let tax = 0.19; // TODO: fix later
4 // 6 months pass. Nobody fixes it.
5}
6
7// Six months later:
8function applyDiscount(price) {
9 // whole file is messy anyway
10 if (price > 100) price = price - 10;
11 return price * 0.95; // double discount?
12}
13
14// Fix it immediately:
15const TAX_RATE = config.get('TAX_RATE');
● 06 / LEADERSHIP

Be a Catalyst
for Change

Don't pitch ideas in Slack and wait. Start it yourself. Show a working proof of concept. People contribute to things they can see and touch, not abstract proposals. Prototypes and demos recruit allies far better than meetings.

⚡
KEY TAKEAWAY
Working software is the most persuasive argument. Build the stone soup others will add ingredients.
user.ts
1// Instead of: "We should adopt TypeScript"
2// in Slack, no traction for 6 months...
3
4// You: convert one module quietly.
5
6interface User {
7 id: number;
8 name: string;
9 email: string;
10}
11
12export function getDisplayName(user: User): string {
13 return `${user.name} <${user.email}>`;
14}
15
16// "Caught 3 bugs. Now they want in."
● 07 / MASTERY

Know Your
Tools Deeply

Pragmatic programmers invest in mastering their tools: editor, shell, debugger, version control, profiler. Not just basic usage shortcuts, power features, scripting. Your tools should be an extension of thought.

⚡
KEY TAKEAWAY
Invest 15 minutes a week learning one new thing about a tool you use daily. In a year, you'll operate at a different level.
terminal
1# Most devs know: git commit, git push
2# Pragmatic programmers also know:
3
4# Rewrite last 5 commits interactively
5git rebase -i HEAD~5
6
7# Find which commit introduced a bug
8git bisect start
9git bisect bad HEAD
10git bisect good v1.2.0
11
12# Stash only staged changes
13git stash push --staged
14
15# Search all commits for a string
16git log -S "calculateDiscount" --source --all
● 08 / RELIABILITY

Design with
Contracts

Every function makes a deal. Make that deal explicit in the code as assertions and checks so when someone breaks the deal, the code fails immediately at the violation site. Not three layers deep in some unrelated crash.

⚡
KEY TAKEAWAY
When you make contracts explicit, bugs surface at the violation site not three layers deep where they're invisible.
math.js
1// No contract: mystery failures
2function divide(a, b) {
3 return a / b; // b could be 0, null, "x"
4}
5
6// With contract
7function divide(a, b) {
8 console.assert(
9 typeof a === 'number' && isFinite(a),
10 'a must be a finite number'
11 );
12 console.assert(
13 typeof b === 'number' && b !== 0,
14 'b must be non-zero'
15 );
16 const result = a / b;
17 console.assert(isFinite(result), 'result finite');
18 return result;
19}
● 09 / PROCESS

Estimate
Everything

Good developers don't freeze when asked "how long will this take?" Break the work into pieces, spot the risky unknowns, and give an honest range. Write down your estimates, then check them when done that feedback loop turns guessing into knowing.

⚡
KEY TAKEAWAY
Estimates given as ranges with named risks are honest. Single numbers are wishful thinking.
estimates.md
1// Bad: "It'll take 2 weeks" (pure guess)
2
3// Good: optimistic / likely / pessimistic
4
5Feature: Payment Integration
6 Research Stripe API 1d / 1d / 2d
7 Checkout flow 2d / 3d / 5d
8 Handle webhooks 1d / 2d / 3d
9 Error handling 1d / 2d / 4d
10 Write tests 1d / 2d / 3d
11 Code review 0.5d / 1d / 2d
12
13// Likely: ~11 days (2.5 weeks)
14// Risk: Stripe sandbox, unclear requirements
15// Flag these never hide them.
● 10 / DEBUGGING

Rubber Ducking

When you're stuck, explain the problem out loud to a colleague, yourself, or a rubber duck. The act of narrating forces your brain to slow down and confront every assumption one by one. The insight comes from you, not the duck.

⚡
KEY TAKEAWAY
The duck never says a word. Keep something on your desk to talk to it sounds silly until it saves you two hours.
debug.js
1// You're stuck. Something's wrong.
2
3// Out loud:
4// "So here I fetch the user by ID..."
5// "...then I pass their ID to buildProfile()..."
6// "...wait I'm passing user.id but
7// the function expects user.profileId.
8// Those aren't the same field."
9
10// Fixed.
11
12// The duck didn't say a word.
13// Explaining forced you to confront
14// every assumption, one by one.
15
16// Keep a duck on your desk. 🦆
